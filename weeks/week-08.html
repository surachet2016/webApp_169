1. HTTP เป็น Stateless — ปัญหาคืออะไร?

HTTP (HyperText Transfer Protocol) เป็นโปรโตคอลแบบ Stateless หมายความว่าทุก Request เป็นอิสระต่อกัน เซิร์ฟเวอร์จำไม่ได้ว่า Request ก่อนหน้านี้มาจากใคร เหมือนกับการโทรศัพท์ที่เมื่อวางสายแล้วทุกอย่างก็จบ ครั้งต่อไปที่โทรมาต้องแนะนำตัวใหม่ทั้งหมด

ปัญหานี้ทำให้ระบบเว็บที่ต้องการ "จำ" ผู้ใช้ เช่น ระบบ Login, ตะกร้าสินค้า, การตั้งค่าส่วนตัว ทำงานไม่ได้หากไม่มีกลไกพิเศษ เพราะทุกครั้งที่กด Link หรือ Refresh หน้า เซิร์ฟเวอร์มองว่าเป็นผู้ใช้ใหม่ที่ไม่รู้จักทุกครั้ง

แนวทางแก้ปัญหาหลักมี 2 วิธี: Session (เก็บข้อมูลบนเซิร์ฟเวอร์) และ Cookie (เก็บข้อมูลในเบราว์เซอร์ของผู้ใช้) ทั้งสองวิธีทำงานร่วมกัน โดย Session ใช้ Cookie เก็บ Session ID เพื่อเชื่อมผู้ใช้กับข้อมูลที่อยู่บนเซิร์ฟเวอร์

หมายเหตุ

HTTP/2 และ HTTP/3 ยังคงเป็น Stateless อยู่เหมือนเดิม แม้จะมีการปรับปรุงด้านประสิทธิภาพ การจัดการ State ยังคงต้องใช้ Session หรือ Cookie เหมือนเดิม

2. Session — จดจำสถานะผู้ใช้ฝั่งเซิร์ฟเวอร์

Session ทำงานโดยเซิร์ฟเวอร์สร้าง Session ID ที่ไม่ซ้ำกัน (เช่น abc123xyz789) แล้วส่ง Cookie ชื่อ PHPSESSID ไปเก็บในเบราว์เซอร์ของผู้ใช้ ทุกครั้งที่ผู้ใช้ส่ง Request มา เบราว์เซอร์จะส่ง Cookie นี้ไปด้วยอัตโนมัติ เซิร์ฟเวอร์จึงสามารถระบุตัวผู้ใช้และดึงข้อมูล Session ได้

ใน PHP การเริ่มใช้ Session ต้องเรียก session_start() เป็นบรรทัดแรกสุด ก่อน มีการ Output ใด ๆ (รวมถึง whitespace) มิฉะนั้นจะเกิด Error "Cannot send session cache limiter" ข้อมูล Session เก็บใน $_SESSION ซึ่งเป็น Superglobal Array ที่เข้าถึงได้ทุกหน้า และข้อมูลเก็บบนเซิร์ฟเวอร์ (โดยปกติในโฟลเดอร์ /tmp)

การทำลาย Session ทำได้ด้วย session_destroy() แต่ต้องล้าง $_SESSION ด้วย $_SESSION = [] ก่อน และลบ Cookie ด้วย setcookie(session_name(), '', time() - 3600) เพื่อให้ Log Out สมบูรณ์

  เบราว์เซอร์                      PHP Server
  ──────────                      ──────────
  Request: GET /cart.php
  Cookie: PHPSESSID=abc123  ────► session_start()
                                   ดึงข้อมูล Session ID: abc123
                                   $_SESSION = ['cart' => [...], 'user_id' => 5]
                                          │
                             ◄────  ส่ง HTML กลับพร้อมข้อมูลตะกร้า

3. Cookie — จัดเก็บข้อมูลฝั่ง Client

Cookie เป็นข้อมูลขนาดเล็กที่เซิร์ฟเวอร์ส่งให้เบราว์เซอร์เก็บไว้ และเบราว์เซอร์จะส่งกลับมาทุกครั้งที่ Request ไปยัง Domain เดิม ใน PHP สร้าง Cookie ด้วย setcookie(name, value, expire, path, domain, secure, httponly)

Parameter ที่สำคัญ: expire คือเวลาหมดอายุเป็น Unix Timestamp เช่น time() + 86400 * 7 = 7 วัน, secure = true หมายถึงส่ง Cookie ผ่าน HTTPS เท่านั้น, httponly = true หมายถึง JavaScript ไม่สามารถอ่าน Cookie นี้ได้ (ป้องกัน XSS ขโมย Cookie)

Cookie มีข้อจำกัดด้านขนาด (ประมาณ 4 KB ต่อ Cookie) และเบราว์เซอร์สามารถลบหรือปิดการรับ Cookie ได้ ดังนั้นอย่าเก็บข้อมูลสำคัญใน Cookie ใช้เก็บเพียงข้อมูลที่ไม่เป็นความลับ เช่น การตั้งค่า Theme หรือภาษา

เคล็ดลับ

ตั้งค่า httponly: true และ secure: true เสมอสำหรับ Cookie ที่เกี่ยวกับ Authentication เพื่อป้องกัน JavaScript ขโมย Session ID

4. ความแตกต่างระหว่าง Session และ Cookie

การเลือกใช้ Session หรือ Cookie ขึ้นอยู่กับประเภทของข้อมูลและระดับความปลอดภัยที่ต้องการ ดูตารางเปรียบเทียบด้านล่าง:

  คุณสมบัติ          Session                      Cookie
  ─────────────────────────────────────────────────────────
  ที่เก็บข้อมูล      เซิร์ฟเวอร์                  เบราว์เซอร์ผู้ใช้
  ความปลอดภัย       สูงกว่า (ไม่ส่งข้อมูลจริง)    ต่ำกว่า (ข้อมูลอยู่ที่ Client)
  อายุการใช้งาน      จนปิด Browser หรือ Timeout   กำหนดได้ (วัน/เดือน/ปี)
  ขนาดข้อมูล        ไม่จำกัด (เก็บในไฟล์เซิร์ฟเวอร์) จำกัด ~4 KB ต่อ Cookie
  การใช้งาน         Login, Cart, User State       Remember Me, Preferences
  การเข้าถึงด้วย JS  ไม่ได้โดยตรง                 ได้ (เว้นแต่ httponly)

5. ความปลอดภัยของ Session

Session Fixation Attack คือการที่ผู้โจมตีส่ง Session ID ที่รู้จักไปให้เหยื่อ หากเหยื่อ Login ด้วย Session ID นั้น ผู้โจมตีก็สามารถเข้าถึง Session ได้ วิธีป้องกันคือเรียก session_regenerate_id(true) ทันทีหลัง Login สำเร็จ เพื่อสร้าง Session ID ใหม่และลบ ID เก่าทิ้ง

ควรกำหนด Session Timeout เพื่อ Log Out อัตโนมัติเมื่อไม่มีการใช้งานนาน เช่น 30 นาที โดยเก็บเวลา Last Activity ไว้ใน $_SESSION['last_activity'] แล้วเช็คทุก Request ว่าผ่านไปนานแค่ไหน นอกจากนี้ควรใช้ HTTPS สำหรับทุก Session ที่เกี่ยวกับ Authentication

ข้อควรระวัง

อย่าเก็บรหัสผ่านหรือข้อมูลบัตรเครดิตใน $_SESSION แม้ Session จะปลอดภัยกว่า Cookie แต่หากเซิร์ฟเวอร์ถูกเข้าถึง ข้อมูลเหล่านั้นก็รั่วได้

🧪 ปฏิบัติการ Lab 7 — Session & Cookie

ทดลองใช้ Session และ Cookie เพื่อจัดการสถานะผู้ใช้ สร้างตะกร้าสินค้า และป้องกันหน้าเว็บ

1
ทดลอง Session พื้นฐาน

สร้าง 3 ไฟล์เพื่อทดสอบ Session: session_set.php (ตั้งค่า Session หลายตัว), session_read.php (อ่านและแสดง Session), session_destroy.php (ทำลาย Session ทั้งหมด)

ใน session_set.php ให้ตั้ง: $_SESSION['name'] = 'นักศึกษา', $_SESSION['score'] = 100, $_SESSION['login_time'] = date('H:i:s') จากนั้นเปิด Developer Tools → Application → Cookies เพื่อดู PHPSESSID

เกณฑ์การผ่าน

session_read.php แสดงค่าที่ตั้งไว้ได้ถูกต้องทั้ง 3 ค่า / เห็น PHPSESSID ใน Browser DevTools / session_destroy.php ทำลาย Session แล้ว session_read.php แสดงว่าไม่มีข้อมูล

คำแนะนำ

เรียก session_start() เป็นบรรทัดแรกสุดของทุกไฟล์ที่ใช้ Session ก่อน echo หรือ HTML ใด ๆ มิฉะนั้นจะเกิด Warning "Cannot send session cache limiter"

2
สร้างตะกร้าสินค้าด้วย Session

สร้างระบบตะกร้าสินค้า 2 หน้า: products.php (แสดงรายการสินค้าพร้อมปุ่ม "เพิ่มลงตะกร้า") และ cart.php (แสดงสินค้าในตะกร้า ยอดรวม และปุ่ม "ล้างตะกร้า")

เมื่อกด "เพิ่มลงตะกร้า" หลาย ๆ ครั้งสำหรับสินค้าชิ้นเดิม ให้เพิ่มจำนวน (qty) แทนที่จะเพิ่มรายการใหม่ ข้อมูลตะกร้าต้องคงอยู่เมื่อ Refresh หน้า

เกณฑ์การผ่าน

เพิ่มสินค้า → เห็นใน cart.php / กด Refresh → ตะกร้ายังอยู่ / เพิ่มสินค้าชิ้นเดิม 3 ครั้ง → qty = 3 ไม่ใช่ 3 rows / ล้างตะกร้า → ตะกร้าว่าง

คำแนะนำ

ใช้ Product ID เป็น Key ของ $_SESSION['cart'] เช่น $_SESSION['cart'][$productId]['qty']++ และใช้ PRG Pattern หลัง POST เสมอเพื่อป้องกัน Double Submit

3
นับจำนวนการเข้าชมหน้าด้วย Session

สร้าง counter.php ที่นับจำนวนครั้งที่ผู้ใช้เข้าชมหน้านั้นใน Session ปัจจุบัน แสดงจำนวนและเวลาที่เริ่ม Session แรก และมีปุ่ม "รีเซ็ต" ที่ทำลาย Session และเริ่มนับใหม่

แสดงข้อมูล Session เพิ่มเติม เช่น Session ID, เวลาที่เหลือก่อนหมดอายุ (session.gc_maxlifetime)

เกณฑ์การผ่าน

Refresh หน้าหลาย ๆ ครั้ง → ตัวเลขเพิ่มขึ้น / กด "รีเซ็ต" → กลับเป็น 1 และ Session ID เปลี่ยน / ปิด Browser แล้วเปิดใหม่ → นับใหม่จาก 1

คำแนะนำ

ใช้ $_SESSION['visit_count'] = ($_SESSION['visit_count'] ?? 0) + 1 และ $_SESSION['first_visit'] = $_SESSION['first_visit'] ?? date('H:i:s') เพื่อบันทึกเวลาแรก

4
ทดลอง Cookie — จดจำชื่อผู้ใช้

สร้าง remember.php ที่มีช่อง input ชื่อ และ Checkbox "จดจำชื่อของฉัน 7 วัน" เมื่อ Submit และติ๊ก Checkbox ให้บันทึกชื่อใน Cookie อายุ 7 วัน เมื่อกลับมาหน้านี้อีกครั้งให้แสดงชื่อที่จำไว้อัตโนมัติ

มีปุ่ม "ลืมชื่อฉัน" ที่ลบ Cookie โดยตั้ง Expire เป็นอดีต (time() - 3600) และแสดงวันหมดอายุของ Cookie

เกณฑ์การผ่าน

กรอกชื่อ + ติ๊ก "จดจำ" → ปิด Browser → เปิดใหม่ → ชื่อยังอยู่ / กด "ลืมชื่อ" → เปิดใหม่ → ไม่มีชื่อ / DevTools Cookies แสดง Cookie อายุ 7 วัน

คำแนะนำ

Cookie ต้อง set ก่อนมี Output HTML ใด ๆ เหมือน session_start() ใช้ isset($_COOKIE['username']) ตรวจสอบว่ามี Cookie อยู่ก่อนอ่าน

5
ป้องกันหน้าด้วย Session

สร้าง Mock Login System: fake_login.php (ฟอร์ม Username + Password อย่างง่าย — ตรวจสอบแค่ว่าไม่ว่างเปล่า ไม่ต้องต่อ DB) และ protected.php (หน้าที่เข้าได้เฉพาะผู้ที่ "Login" แล้วเท่านั้น)

ถ้าพยายามเปิด protected.php โดยตรง (ไม่ได้ Login) ต้อง Redirect ไป fake_login.php อัตโนมัติ เมื่อ Login แล้วให้ set $_SESSION['logged_in'] = true และมีปุ่ม "Logout" ที่ทำลาย Session

เกณฑ์การผ่าน

เปิด protected.php โดยตรง → Redirect ไป fake_login.php / Login → เห็นเนื้อหาของ protected.php / กด Logout → กลับไป protected.php → Redirect ไป Login อีกครั้ง

คำแนะนำ

ใส่โค้ดตรวจสอบ Session ที่ต้น protected.php ก่อนโค้ดอื่น: if (!isset($_SESSION['logged_in'])) { header('Location: fake_login.php'); exit; }
