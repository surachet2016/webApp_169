ภาพรวมโครงงาน

ถึงเวลาที่น่าตื่นเต้นที่สุดของรายวิชา — โครงงานจริง! นักศึกษาจะทำงานเป็นทีม ทีมละ 3-4 คน เพื่อสร้างเว็บแอปพลิเคชันธุรกิจจริงๆ ขึ้นมา โดยนำทุกสิ่งที่เรียนมาตลอด 12 สัปดาห์มาประยุกต์ใช้:

  • HTML, CSS, JavaScript — สร้าง Interface ที่ใช้งานง่าย
  • PHP — ประมวลผลฝั่ง Server
  • MySQL — จัดเก็บและจัดการข้อมูล
  • CRUD — Create, Read, Update, Delete ข้อมูล
  • Authentication — ระบบเข้าสู่ระบบและสิทธิ์การใช้งาน
  • File Upload — อัปโหลดรูปภาพหรือเอกสาร
  • REST API (optional) — เพิ่มคะแนนถ้ามีการใช้ API

ไทม์ไลน์: สัปดาห์ที่ 13 — วางแผนและเริ่มพัฒนา | สัปดาห์ที่ 14 — นำเสนอโครงงาน

หมายเหตุ

โครงงานต้องเป็นระบบที่มีการโต้ตอบจริง (ไม่ใช่แค่หน้าเว็บแสดงข้อมูลอย่างเดียว) ต้องมีฐานข้อมูล MySQL อย่างน้อย 3 ตาราง มีระบบ Login และมี CRUD อย่างน้อย 2 ชุด โปรเจกต์ที่ซับซ้อนกว่าจะได้คะแนนสูงกว่า แต่ความถูกต้องและการทำงานได้จริงสำคัญกว่าจำนวน Feature

การวิเคราะห์ความต้องการ (Requirements Analysis)

ก่อนเขียนโค้ดแม้แต่บรรทัดเดียว ทีมต้องตอบคำถามเหล่านี้ให้ชัดเจน:

  • ปัญหาคืออะไร? — ระบบนี้แก้ปัญหาอะไรในโลกจริง?
  • ใครคือผู้ใช้? — Admin, ลูกค้า, พนักงาน, สมาชิก?
  • ต้องการ Feature อะไรบ้าง? — แยกระหว่าง "ต้องมี" กับ "อยากมี"

ใช้รูปแบบ User Stories เพื่ออธิบาย Requirement อย่างเป็นระบบ: 

รูปแบบ User Story:
  "ในฐานะ [ประเภทผู้ใช้] ฉันต้องการ [สิ่งที่ต้องการทำ] เพื่อที่จะ [ประโยชน์ที่ได้รับ]"

ตัวอย่าง (ระบบหอพัก):
  ┌─────────────────────────────────────────────────────────────────────────┐
  │ ในฐานะ "ผู้เช่าห้อง" ฉันต้องการ "ดูรายการค่าใช้จ่ายประจำเดือน"       │
  │ เพื่อที่จะ "วางแผนการชำระเงินได้ล่วงหน้า"                             │
  ├─────────────────────────────────────────────────────────────────────────┤
  │ ในฐานะ "ผู้ดูแลระบบ" ฉันต้องการ "เพิ่ม/แก้ไข/ลบข้อมูลห้องพัก"       │
  │ เพื่อที่จะ "จัดการห้องว่างและห้องที่มีผู้เช่าได้อย่างมีประสิทธิภาพ"  │
  ├─────────────────────────────────────────────────────────────────────────┤
  │ ในฐานะ "ผู้เช่าห้อง" ฉันต้องการ "แจ้งซ่อมแซมห้องผ่านระบบออนไลน์"    │
  │ เพื่อที่จะ "ไม่ต้องเดินไปหาผู้ดูแลด้วยตนเอง"                         │
  └─────────────────────────────────────────────────────────────────────────┘
เคล็ดลับ

แบ่ง Requirements เป็น 2 ระดับ: Must Have (จำเป็น — ทำให้ได้ใน 2 สัปดาห์) และ Nice to Have (ถ้ามีเวลาเหลือ) เพื่อป้องกัน Scope Creep ที่ทำให้โปรเจกต์ไม่เสร็จทันเวลา

การออกแบบระบบ (System Design)

เมื่อรู้ Requirement แล้ว ต้องออกแบบระบบก่อนเริ่มเขียนโค้ด:

1. ER Diagram (Entity-Relationship Diagram) — แผนภาพที่แสดงตารางในฐานข้อมูล และความสัมพันธ์ระหว่างตาราง ประกอบด้วย Entities (ตาราง), Attributes (คอลัมน์), และ Relationships (FK) 

ตัวอย่าง ER Diagram (ระบบขายสินค้า):

[users]              [products]           [categories]
──────               ─────────            ────────────
id (PK)              id (PK)              id (PK)
username             name                 name
email                price                description
password             stock
role                 image
created_at           category_id (FK)──────►[categories.id]

[orders]             [order_items]
────────             ─────────────
id (PK)              id (PK)
user_id (FK)─────────►[users.id]
status               order_id (FK)────────►[orders.id]
total_price          product_id (FK)──────►[products.id]
created_at           quantity
                     unit_price

2. Page Flow Diagram / Wireframe — แผนผังที่แสดงว่ามีหน้าอะไรบ้าง และผู้ใช้ไหลจากหน้าหนึ่งไปยังอีกหน้าอย่างไร 

ตัวอย่าง Page Flow (ระบบขายสินค้า):

[หน้าหลัก] ──► [รายการสินค้า] ──► [รายละเอียดสินค้า]
                                            │
                                     [ตะกร้าสินค้า]
                                            │
                        ┌──────────────[Login]
                        ▼
                  [ชำระเงิน] ──► [ยืนยันคำสั่งซื้อ]

[Admin Panel] ──► [จัดการสินค้า] (CRUD)
              └──► [จัดการคำสั่งซื้อ]
              └──► [จัดการผู้ใช้]
ข้อควรระวัง

อย่าข้ามขั้นตอนการออกแบบ หลายทีมรีบเขียนโค้ดเลยโดยไม่มี ER Diagram หรือ Wireframe แล้วพบว่าต้องแก้โครงสร้างฐานข้อมูลกลางคัน ซึ่งทำให้เสียเวลามากกว่าการวางแผนตั้งแต่ต้น ใช้เวลาอย่างน้อย 1-2 ชั่วโมงวางแผนให้ดีก่อน

การแบ่งงานในทีม (Team Work)

การทำงานเป็นทีมที่ดีต้องมีการจัดการที่ชัดเจน:

  • Team Leader — ประสานงาน ติดตามความคืบหน้า แก้ปัญหาความขัดแย้ง และเป็นตัวแทนทีมในการนำเสนอ
  • Frontend Developer — รับผิดชอบ HTML, CSS, JavaScript ออกแบบและสร้าง UI ที่ใช้งานง่าย
  • Backend Developer — รับผิดชอบ PHP Logic, API Endpoints การ Validate ข้อมูล และความปลอดภัย
  • Database Designer — ออกแบบโครงสร้างฐานข้อมูล สร้างตาราง เขียน SQL Queries ดูแลความสมบูรณ์ของข้อมูล

แนะนำวิธีติดตามความคืบหน้า:

  • Daily Standup (15 นาที) — ทุกคนบอก: ทำอะไรเสร็จแล้ว, กำลังทำอะไร, มีปัญหาอะไร
  • Task Board — ใช้กระดาษหรือ Google Sheet แบ่ง Tasks เป็น To Do / In Progress / Done
  • Git (optional) — ใช้ Git สำหรับ Version Control ป้องกัน Code ชนกัน

ไอเดียโครงงานที่แนะนำ

เลือกโครงงานที่ทีมสนใจและสามารถทำเสร็จได้ใน 2 สัปดาห์ ต่อไปนี้คือไอเดียที่เหมาะสม:

  • ระบบขายสินค้าออนไลน์ (E-Commerce) — จัดการสินค้า (ชื่อ ราคา รูป สต็อก), ตะกร้าสินค้า, ระบบคำสั่งซื้อ, หน้า Admin สำหรับจัดการสินค้าและออเดอร์
  • ระบบจัดการหอพัก (Dormitory Management) — ข้อมูลห้อง ผู้เช่า ค่าเช่า/ค่าน้ำ/ค่าไฟ, แจ้งซ่อม, รายงานรายรับ-รายจ่าย
  • ระบบลงทะเบียนกิจกรรม (Event Registration) — สร้างกิจกรรม, ลงทะเบียนเข้าร่วม, จัดการผู้เข้าร่วม, แสดง QR Code, รายงาน
  • ระบบห้องสมุด (Library Management) — จัดการหนังสือ, ยืม-คืน, แจ้งเตือนหนังสือเกินกำหนด, ค้นหาและจอง
  • เว็บไซต์ข่าวสารชุมชน (Community News) — โพสต์ข่าว/บทความ, หมวดหมู่, ความคิดเห็น, ระบบสมาชิก, Admin จัดการเนื้อหา
  • ระบบจองห้องประชุม (Meeting Room Booking) — ห้องประชุมหลายห้อง, จองช่วงเวลา, ตรวจสอบห้องว่าง, ยกเลิกการจอง, Dashboard
เคล็ดลับ

เลือกโครงงานที่ทีมเข้าใจ Domain (ความรู้ด้านนั้น) ดี เพราะจะทำให้การออกแบบฐานข้อมูล และ Feature ถูกต้องและสมจริงกว่า เช่น ถ้าเคยอยู่หอพักก็ทำระบบหอพัก ถ้าชอบขายของก็ทำ E-Commerce

database.sql (ตัวอย่างระบบ E-Commerce) SQL 
-- database.sql — โครงสร้างฐานข้อมูลสำหรับระบบขายสินค้า
-- นำเข้าด้วย: phpMyAdmin → Import หรือ mysql -u root -p < database.sql

CREATE DATABASE IF NOT EXISTS `ecommerce_db`
  DEFAULT CHARACTER SET utf8mb4
  DEFAULT COLLATE utf8mb4_unicode_ci;

USE `ecommerce_db`;

-- ─── ตารางผู้ใช้ ───────────────────────────────────────────
CREATE TABLE `users` (
  `id`         INT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
  `username`   VARCHAR(50)  NOT NULL UNIQUE,
  `email`      VARCHAR(100) NOT NULL UNIQUE,
  `password`   VARCHAR(255) NOT NULL,             -- bcrypt hash
  `role`       ENUM('customer','admin') DEFAULT 'customer',
  `created_at` TIMESTAMP DEFAULT CURRENT_TIMESTAMP
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

-- ─── ตารางหมวดหมู่สินค้า ───────────────────────────────────
CREATE TABLE `categories` (
  `id`          INT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
  `name`        VARCHAR(100) NOT NULL,
  `description` TEXT,
  `created_at`  TIMESTAMP DEFAULT CURRENT_TIMESTAMP
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

-- ─── ตารางสินค้า ────────────────────────────────────────────
CREATE TABLE `products` (
  `id`          INT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
  `category_id` INT UNSIGNED,
  `name`        VARCHAR(200) NOT NULL,
  `description` TEXT,
  `price`       DECIMAL(10,2) NOT NULL DEFAULT 0.00,
  `stock`       INT NOT NULL DEFAULT 0,
  `image`       VARCHAR(255),
  `active`      TINYINT(1) NOT NULL DEFAULT 1,
  `created_at`  TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  FOREIGN KEY (`category_id`) REFERENCES `categories`(`id`) ON DELETE SET NULL
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

-- ─── ตารางคำสั่งซื้อ ──────────────────────────────────────
CREATE TABLE `orders` (
  `id`          INT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
  `user_id`     INT UNSIGNED NOT NULL,
  `status`      ENUM('pending','processing','shipped','delivered','cancelled')
                DEFAULT 'pending',
  `total_price` DECIMAL(10,2) NOT NULL,
  `note`        TEXT,
  `created_at`  TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  FOREIGN KEY (`user_id`) REFERENCES `users`(`id`) ON DELETE CASCADE
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

-- ─── ตารางรายการสินค้าในคำสั่งซื้อ ───────────────────────
CREATE TABLE `order_items` (
  `id`         INT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
  `order_id`   INT UNSIGNED NOT NULL,
  `product_id` INT UNSIGNED NOT NULL,
  `quantity`   INT NOT NULL DEFAULT 1,
  `unit_price` DECIMAL(10,2) NOT NULL,  -- ราคา ณ เวลาที่สั่งซื้อ
  FOREIGN KEY (`order_id`)   REFERENCES `orders`(`id`)   ON DELETE CASCADE,
  FOREIGN KEY (`product_id`) REFERENCES `products`(`id`) ON DELETE CASCADE
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

Git Workflow สำหรับทีม Bash 
# ─── Team Leader: ตั้งค่า Repository ─────────────────────────
git init
git add .
git commit -m "initial project structure"
# สร้าง repo บน GitHub แล้ว push
git remote add origin https://github.com/username/project-name.git
git push -u origin main

# ─── สมาชิกอื่น: Clone โปรเจกต์ ──────────────────────────────
git clone https://github.com/username/project-name.git
cd project-name

# ─── Workflow ทุกวัน ───────────────────────────────────────────
# 1. ดึงโค้ดล่าสุดจากทีม
git pull origin main

# 2. สร้าง Branch ใหม่สำหรับ Feature ที่กำลังทำ
git checkout -b feature/login-page

# 3. เขียนโค้ด...

# 4. บันทึกงาน
git add login.php includes/auth.php
git commit -m "add user login with session management"

# 5. Push ขึ้น GitHub
git push origin feature/login-page

# 6. สร้าง Pull Request บน GitHub เพื่อให้ Team Leader Review

# ─── ไฟล์ .gitignore (ควรสร้างก่อน push แรก) ─────────────────
# สร้างไฟล์ .gitignore:
# config/config.php     (ซ่อนรหัสผ่าน DB)
# uploads/              (ไม่ push ไฟล์ที่ user upload)
# .env                  (ถ้าใช้)
# *.log
# .DS_Store             (Mac)
# Thumbs.db             (Windows)

# ─── แก้ Conflict กรณีโค้ดชนกัน ──────────────────────────────
git pull origin main        # ดึงโค้ดล่าสุด
# ถ้ามี conflict: เปิดไฟล์ที่ conflict แก้ไข แล้ว:
git add conflicted-file.php
git commit -m "resolve merge conflict in conflicted-file.php"
งานที่ต้องส่ง (Deliverables) — สัปดาห์ที่ 13
  • 1. เอกสารวิเคราะห์ความต้องการ (Requirements Document) — รายชื่อสมาชิกทีม, ชื่อโปรเจกต์, ปัญหาที่แก้, รายการ User Stories อย่างน้อย 8 ข้อ
  • 2. ER Diagram ของฐานข้อมูล — แสดงตารางทั้งหมด (อย่างน้อย 3 ตาราง), คอลัมน์สำคัญ, และ Foreign Keys ที่เชื่อมตารางกัน
  • 3. Wireframe/Mockup อย่างน้อย 3 หน้า — วาดด้วยมือ, PowerPoint, หรือโปรแกรม เช่น Figma/Canva ให้เห็น Layout และ Navigation
  • 4. แผนการแบ่งงานในทีม — ตารางแสดงชื่อสมาชิก, Role, Tasks ที่รับผิดชอบ, และ Timeline (จะเสร็จเมื่อไร)
