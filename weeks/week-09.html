1. ระบบสมาชิก (Membership System) คืออะไร?

ระบบสมาชิกเป็นหนึ่งในคุณสมบัติหลักของเว็บแอปพลิเคชันเกือบทุกประเภท ตั้งแต่ร้านค้าออนไลน์ โซเชียลมีเดีย ไปจนถึงระบบบริหารจัดการภายในองค์กร ระบบสมาชิกมีองค์ประกอบหลักได้แก่: Registration (สมัครสมาชิก), Login (เข้าสู่ระบบ), Logout (ออกจากระบบ), Profile (จัดการข้อมูลส่วนตัว) และ Password Reset (รีเซ็ตรหัสผ่าน)

การออกแบบระบบสมาชิกที่ดีต้องคำนึงถึง: ความปลอดภัยของข้อมูล (โดยเฉพาะรหัสผ่าน), ประสบการณ์ผู้ใช้ที่ราบรื่น, การจัดการ Error อย่างเหมาะสม และการป้องกันการโจมตีประเภทต่าง ๆ เช่น Brute Force, SQL Injection และ XSS

สำหรับหลักสูตรนี้เราจะเน้นสร้างระบบ Registration และ Login พื้นฐานที่ปลอดภัย โดยใช้ PDO สำหรับฐานข้อมูล, password_hash() สำหรับเข้ารหัสรหัสผ่าน และ Session สำหรับจัดการสถานะ Login ซึ่งเป็นมาตรฐานที่ใช้ในระบบงานจริง

2. การเข้ารหัสรหัสผ่าน (Password Hashing)

ห้ามเก็บรหัสผ่านเป็น Plain Text โดยเด็ดขาด หากฐานข้อมูลถูกขโมย รหัสผ่านของผู้ใช้ทุกคนจะรั่วทันที ในอดีตนักพัฒนาบางคนใช้ MD5 หรือ SHA1 ซึ่งเป็น Hash Function แต่ปัจจุบันสองอย่างนี้ถือว่า ไม่ปลอดภัย เพราะมี Rainbow Table และ GPU สามารถ Crack ได้เร็วมาก

PHP ให้ฟังก์ชัน password_hash($password, PASSWORD_BCRYPT) ที่ใช้อัลกอริทึม bcrypt ซึ่งออกแบบมาให้ช้าโดยเจตนา (ต่างจาก MD5 ที่เร็ว) เพื่อทำให้การ Brute Force ยากและใช้เวลานาน bcrypt ยังเพิ่ม Salt โดยอัตโนมัติ ทำให้รหัสผ่านเดียวกันได้ Hash ต่างกันทุกครั้งที่ Hash

การตรวจสอบรหัสผ่านใช้ password_verify($input, $hash) ซึ่งจะเปรียบเทียบ Input กับ Hash ที่เก็บในฐานข้อมูล ถ้าตรงกันคืนค่า true ไม่ต้อง Hash Input ก่อน เพราะฟังก์ชันทำให้เอง

ข้อควรระวัง

อย่าใช้ MD5 หรือ SHA1 สำหรับรหัสผ่าน! ใช้เฉพาะ password_hash() ที่มาพร้อม PHP 5.5+ เท่านั้น Column รหัสผ่านในฐานข้อมูลต้องเป็น VARCHAR(255) เพราะ bcrypt Hash มีความยาว 60 ตัวอักษร

  password_hash('mySecret123', PASSWORD_BCRYPT)
  ↓
  '$2y$10$abc...xyz'  (60 ตัวอักษร — เก็บในฐานข้อมูล)

  password_verify('mySecret123', '$2y$10$abc...xyz')
  ↓  true ✓

  password_verify('wrongPassword', '$2y$10$abc...xyz')
  ↓  false ✗

3. ระบบ Login และ Logout

กระบวนการ Login ที่ถูกต้องมีขั้นตอนดังนี้: (1) รับ Email/Username และ Password จากฟอร์ม, (2) ค้นหาผู้ใช้จากฐานข้อมูลด้วย Email โดยใช้ Prepared Statement, (3) ตรวจสอบว่าพบ User หรือไม่, (4) ใช้ password_verify() ตรวจสอบรหัสผ่าน, (5) หากถูกต้อง เรียก session_regenerate_id(true) ก่อนตั้ง Session, (6) ตั้งค่า Session Variables, (7) Redirect ไปหน้า Dashboard

สิ่งสำคัญที่ต้องระวังในข้อความ Error: อย่าบอกว่า Email ไม่มีในระบบหรือรหัสผ่านผิด แยกกัน เพราะจะทำให้ผู้โจมตีรู้ว่า Email นั้นมีอยู่ในระบบ ควรแสดงข้อความเดียวกันเสมอ เช่น "อีเมลหรือรหัสผ่านไม่ถูกต้อง"

การ Logout ต้องทำ 3 อย่าง: (1) $_SESSION = [] ล้าง Array, (2) ลบ Session Cookie ด้วย setcookie(session_name(), '', time()-3600), (3) session_destroy() ทำลาย Session บนเซิร์ฟเวอร์ จากนั้น Redirect ไปหน้า Login

4. ความปลอดภัยพื้นฐานของระบบ

SQL Injection คือการที่ผู้โจมตีแทรก SQL Code ผ่านช่อง Input เช่น ใส่ ' OR '1'='1 เป็น Password เพื่อ Bypass การ Login ป้องกันได้ด้วยการใช้ PDO Prepared Statement ทุกครั้งที่รับ Input จากผู้ใช้ ห้ามใช้ String Concatenation สร้าง SQL Query

XSS (Cross-Site Scripting) คือการแทรก JavaScript ผ่าน Input เพื่อขโมย Cookie หรือ Session ป้องกันด้วย htmlspecialchars() ทุกครั้งก่อนแสดงข้อมูลใน HTML

CSRF (Cross-Site Request Forgery) คือการหลอกให้ผู้ใช้ที่ Login อยู่ส่ง Request โดยไม่รู้ตัว ป้องกันด้วยการใส่ CSRF Token ซ่อนไว้ในฟอร์มทุกอัน แล้วตรวจสอบ Token ก่อนประมวลผล สำหรับระบบพื้นฐานในหลักสูตรนี้ยังไม่จำเป็นต้องใช้ แต่ควรทราบว่ามีอยู่

หมายเหตุ

ตรวจสอบ Input ทุกตัวจากผู้ใช้เสมอ ทั้ง $_POST, $_GET, $_COOKIE อย่าเชื่อถือข้อมูลจาก Client โดยไม่ตรวจสอบ แม้จะเป็น Hidden Field ใน Form ก็ตาม

🧪 ปฏิบัติการ Lab 8 — ระบบสมาชิก & Authentication

สร้างระบบสมาชิกสมบูรณ์แบบ ตั้งแต่การสมัคร เข้าสู่ระบบ Dashboard ไปจนถึง Logout พร้อมการเข้ารหัสรหัสผ่านที่ถูกต้อง

1
สร้างตาราง users ในฐานข้อมูล

สร้างตาราง users ในฐานข้อมูล MySQL โดยมี Column ที่ครบถ้วนสำหรับระบบสมาชิก Column password ต้องเป็น VARCHAR(255) เพื่อรองรับ bcrypt Hash

สร้างตาราง users SQL 
CREATE TABLE users (
    id         INT AUTO_INCREMENT PRIMARY KEY,
    username   VARCHAR(50) NOT NULL UNIQUE,
    email      VARCHAR(100) NOT NULL UNIQUE,
    password   VARCHAR(255) NOT NULL,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci;

หลังสร้างตาราง ให้ทดสอบการ Hash รหัสผ่านด้วยโค้ด PHP: echo password_hash('test1234', PASSWORD_BCRYPT); แล้วสังเกตว่า Hash มีความยาวเท่าไรและเริ่มต้นด้วย $2y$10$

เกณฑ์การผ่าน

ตาราง users มีครบทุก Column / ทดสอบ password_hash() แล้ว Hash มีความยาว 60 ตัวอักษรและขึ้นต้นด้วย $2y$10$

คำแนะนำ

ใส่ UNIQUE Constraint ที่ทั้ง username และ email เพื่อป้องกันการสมัครซ้ำในระดับฐานข้อมูล (Defense in Depth) ไม่ใช่แค่ตรวจสอบใน PHP เท่านั้น

2
หน้าสมัครสมาชิก (register.php)

สร้างไฟล์ register.php ที่มีฟอร์ม: ชื่อผู้ใช้, อีเมล, รหัสผ่าน, ยืนยันรหัสผ่าน ต้อง Validate: ทุกฟิลด์กรอกครบ, รูปแบบอีเมลถูกต้อง (filter_var), รหัสผ่านและยืนยันตรงกัน, ชื่อผู้ใช้และอีเมลไม่ซ้ำในระบบ

เมื่อผ่าน Validation ให้ Hash รหัสผ่านด้วย password_hash() และ INSERT ลงตาราง users จากนั้น Redirect ไป login.php

เกณฑ์การผ่าน

สมัครด้วยข้อมูลครบถ้วน → Redirect ไป login.php / สมัครด้วย Email ซ้ำ → Error / รหัสผ่านไม่ตรง → Error / เปิด phpMyAdmin → เห็นข้อมูลในตาราง users โดย Column password เป็น Hash ไม่ใช่ Plain Text

คำแนะนำ

ใช้ filter_var($email, FILTER_VALIDATE_EMAIL) ตรวจสอบรูปแบบอีเมล และ strlen($password) >= 8 ตรวจสอบความยาวรหัสผ่าน ตรวจสอบ Username ซ้ำก่อน INSERT

3
หน้าเข้าสู่ระบบ (login.php)

สร้างไฟล์ login.php ที่มีฟอร์ม: อีเมล และ รหัสผ่าน ใช้ PDO Prepared Statement ค้นหา User ด้วย Email แล้วใช้ password_verify() ตรวจสอบรหัสผ่าน

เมื่อ Login สำเร็จ: เรียก session_regenerate_id(true), ตั้ง Session (user_id, username, logged_in = true), Redirect ไป dashboard.php เมื่อ Login ล้มเหลว: แสดงข้อความ "อีเมลหรือรหัสผ่านไม่ถูกต้อง" เสมอ ไม่ว่าจะผิดที่อีเมลหรือรหัสผ่าน

เกณฑ์การผ่าน

Login ด้วยข้อมูลถูกต้อง → เข้า dashboard.php / Login ด้วยรหัสผ่านผิด → Error Message / Login ด้วย Email ที่ไม่มีในระบบ → Error Message เดียวกัน / DevTools → Session มี user_id

คำแนะนำ

ตรวจสอบทั้ง $user ไม่ว่าง และ password_verify() ในเงื่อนไขเดียวกัน: if ($user && password_verify($password, $user['password'])) เพื่อป้องกัน PHP Error กรณีไม่พบ User

4
หน้า Dashboard (ต้อง Login)

สร้างไฟล์ dashboard.php ที่ตรวจสอบ Session ก่อนแสดงเนื้อหา ถ้ายังไม่ Login ให้ Redirect ไป login.php ทันที แสดง: ข้อความต้อนรับพร้อม Username, อีเมลของผู้ใช้, เวลาที่ Login และปุ่ม Logout

ทดสอบโดยพยายาม Copy URL ของ dashboard.php แล้วเปิดใน Tab ใหม่โดยไม่ Login ต้อง Redirect ไปหน้า Login โดยอัตโนมัติ

เกณฑ์การผ่าน

Login แล้วเปิด dashboard.php → เห็น "ยินดีต้อนรับ [ชื่อ]" / เปิด dashboard.php โดยไม่ Login → Redirect ไป login.php / Username แสดงถูกต้องตรงกับที่สมัคร

คำแนะนำ

ใส่ session_start(); และตรวจสอบ Session ที่บรรทัดแรกสุด ก่อน HTML ใด ๆ จากนั้นค่อยแสดงเนื้อหา สามารถสร้าง auth_check.php แยกไว้แล้ว require เพื่อใช้ซ้ำได้

5
ระบบ Logout สมบูรณ์

สร้างไฟล์ logout.php ที่ทำการ Logout สมบูรณ์แบบ 3 ขั้นตอน: ล้าง $_SESSION = [], ลบ Session Cookie ออกจากเบราว์เซอร์, และ session_destroy() จากนั้น Redirect ไป login.php

ทดสอบ: Login → ไป dashboard.php → กด Logout → พยายามกด Back ของเบราว์เซอร์ → ต้องไม่สามารถเข้า dashboard.php ได้อีก (Redirect กลับ login.php)

เกณฑ์การผ่าน

กด Logout → Redirect ไป login.php / กด Browser Back → Redirect ไป login.php ทันที (ไม่เห็น dashboard) / DevTools Cookies → PHPSESSID หายไปหลัง Logout

คำแนะนำ

ต้องทำครบ 3 ขั้นตอน ขาดขั้นตอนใดขั้นตอนหนึ่งอาจทำให้ Session ยังคงอยู่ หลัง session_destroy() สามารถเพิ่ม Header Cache-Control: no-store เพื่อป้องกัน Browser Cache หน้า Dashboard
